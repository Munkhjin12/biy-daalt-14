# Эргэцүүлэл — REFLECTION.md

## 1. Аль assertion хамгийн их үнэ цэнэтэй санагдсан вэ? Яагаад?

Хамгийн их үнэ цэнэтэй assertion нь **email формат шалгах бизнес дүрэм** байсан.
`pm.expect(d.email).to.be.a('string').and.match(/@/)` — энэ нь зөвхөн "өгөгдөл
байна уу" гэдгийг биш, өгөгдлийн утга нь бодит ертөнцийн дүрмийг хангаж байгаа
эсэхийг шалгадаг. Status code шалгах нь API ажиллаж байгааг харуулна, харин
бизнес дүрэм шалгах нь API зөв ажиллаж байгааг харуулна. Үйлчилгээний contract
буюу "email талбар байх ёстой бөгөөд @ тэмдэгт агуулсан байх ёстой" гэсэн тодорхой
нөхцөлийг тест нь хамгаалдаг тул хамгийн чухал гэж үзсэн.

## 2. Negative test жишээ — яг ямар алдааг олдог вэ?

`GET /posts/999999` request нь байхгүй id-тай хүсэлт явуулж **404 Not Found**
буцаж ирэхийг шалгана. Энэ тест нь "API нь байхгүй нөөцийг зөв зохицуулдаг уу?"
гэсэн асуултад хариулдаг. Хэрэв хөгжүүлэгч error handling буруу хийсэн бол
500 Internal Server Error буцааж болно, эсвэл 200 status-тай хоосон body буцааж
болно. Тест нь яг 404 байхыг шаарддаг тул эдгээр алдааг шууд илрүүлнэ. Мөн
`GET /users/99999` тест нь хэрэглэгчийн модульд ижил алдаа байгаа эсэхийг
тусдаа шалгадаг. Negative test байхгүй бол зөвхөн "хэвийн замыг" шалгаж,
алдааны зам хяналтгүй үлдэнэ.

## 3. Postman-д ажилласан тест Newman-д fail болсон уу? Яагаад?

JSONPlaceholder нь нийтийн API тул environment тохируулга зөв хийгдвэл Postman
болон Newman хоёрт ижил үр дүн гарна. Гэхдээ нэг боломжит ялгаа гарч болно:
Postman-д GUI-р `postId` хувьсагч нь өмнөх session-оос хадгалагдсан байдаг,
харин Newman-д collection яг дарааллаар ажилладаг тул "Happy GET — Бүх нийтлэл
авах" request заавал эхэлж ажиллан `postId`-г environment-д тохируулах ёстой.
Хэрэв collection дараалал өөрчлөгдвөл chain хийсэн request fail болох эрсдэлтэй.
Энэ учраас collection-ийн дараалалд онцгой анхаарч, chain хийх request заавал
өмнөө байх ёстой.

## 4. Token/secret-ыг хэрхэн зохицуулсан вэ?

JSONPlaceholder нь auth шаардахгүй тул энэ лабораторид token хэрэглэгдсэнгүй.
Гэхдээ env файлын соёлыг бүрэн дагасан: `env.dev.json` болон `env.ci.json`
файлуудад бодит нууц огт орхигдоогүй. Хэрэв token шаардагдах API сонгосон бол
`REPLACE_THIS_WITH_REAL_TOKEN` placeholder ашиглан README-д тайлбар бичих байсан.
GitHub Secrets-т `API_TOKEN` нэрээр хадгалж, workflow-д `${{ secrets.API_TOKEN }}`
хэлбэрээр inject хийх байсан. `.gitignore`-д бодит token агуулсан файл нэмэх
нь security-ийн үндсэн алхам юм. "Secret-ыг code-д бичихгүй" дүрэм нь
командаас биш, зан үйлийн соёлоос үүддэг.

## 5. API өөрчлөгдвөл collection-ийн аль хэсэг хамгийн их эвдрэх вэ?

Хамгийн "эмзэг газар" нь **chain хийсэн request-ийн хувьсагч дамжуулалт** болон
**schema assertion-ууд** юм. Хэрэв `/posts` response-ийн field нэр өөрчлөгдвөл
(жишээ нь `userId` → `authorId`) тэр даруй олон тест цуварч fail болно: schema
тест, бизнес дүрэм тест, `userId`-г environment-д тохируулах мөр, дараагийн
`GET /users/{{userId}}` request бүгд унана. Энэ эрсдлийг бууруулахын тулд:
(1) schema validation-ыг нэг газарт Collection Variables-р тодорхойлох;
(2) chain хийх хувьсагчдыг тусдаа Pre-request script-т төвлөрүүлэх;
(3) API versioning-г дагах (`/v1/posts`) тул version upgrade хийхдээ
тестийн collection-ыг хамт шинэчлэх workflow бий болгох нь зүйтэй.
